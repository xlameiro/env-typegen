## Config file

Create `env-typegen.config.mjs` at your project root:

```typescript
import { defineConfig } from "@xlameiro/env-typegen";

export default defineConfig({
  version: 1,
  input: ".env.example",
  output: "env.generated.ts",
  generators: ["typescript", "zod"],
  format: true,
  schemaFile: "env.contract.mjs",
  strict: true,
  diffTargets: [".env", ".env.example", ".env.production"],
  providers: {
    vercel: {
      adapter: "./src/adapters/vercel-adapter.ts",
      projectId: "project_123",
      token: "${VERCEL_TOKEN}",
    },
  },
  environments: {
    development: { source: "local" },
    preview: { source: "vercel", targets: ["preview"] },
    production: { source: "vercel", targets: ["production"] },
  },
  rules: {
    strict: true,
    allowExtra: false,
    requireAll: true,
  },
  plugins: ["./plugins/custom-validator.mjs"],
});
```

For direct Node.js runtime execution, prefer `.mjs` or `.js` config files.
TypeScript config files (`.ts`) require a runtime loader such as `tsx` or `ts-node`.

## Advanced config options

### `providers`

Registers adapter-backed providers for `pull` and governance workflows.

```typescript
export default defineConfig({
  providers: {
    vercel: {
      adapter: "./src/adapters/vercel-adapter.ts",
      projectId: "project_123",
      token: process.env.VERCEL_TOKEN,
    },
  },
});
```

### `environments`

Maps logical environments to source providers and target aliases.

```typescript
export default defineConfig({
  environments: {
    development: { source: "local" },
    preview: { source: "vercel", targets: ["preview"] },
    production: { source: "vercel", targets: ["production"] },
  },
});
```

### `rules`

Defines governance policy defaults globally and by environment.

```typescript
export default defineConfig({
  rules: {
    strict: true,
    allowExtra: false,
    requireAll: true,
    perEnvironment: {
      development: { allowExtra: true },
      production: { strict: true, allowExtra: false, requireAll: true },
    },
  },
});
```

### `schemaMode`

Controls how env-typegen resolves type information when a contract is present.

- `legacy` (default): inference-first behavior with annotation overrides.
- `contract-first`: the contract becomes authoritative; inference is used as fallback only.

```typescript
export default defineConfig({
  input: ".env.example",
  schemaFile: "env.contract.mjs",
  schemaMode: "contract-first",
});
```

### `inferenceRules`

Adds custom type inference rules before built-in rules. Rules are evaluated in ascending `priority` order (lower number runs first).

```typescript
export default defineConfig({
  input: ".env.example",
  inferenceRules: [
    {
      type: "string",
      priority: 5,
      match(key, value) {
        return key.endsWith("_TOKEN") && value.length > 0;
      },
    },
  ],
});
```

## Practical presets

### Generation-focused preset

```typescript
export default defineConfig({
  input: ".env.example",
  output: "env.generated.ts",
  generators: ["typescript", "zod"],
  format: true,
});
```

### Governance-focused preset

```typescript
export default defineConfig({
  schemaFile: "env.contract.mjs",
  strict: true,
  diffTargets: [".env", ".env.staging", ".env.production"],
});
```

### Monorepo-friendly preset

```typescript
export default defineConfig({
  input: "apps/web/.env.example",
  output: "apps/web/src/env.generated.ts",
  schemaFile: "config/env.contract.mjs",
  diffTargets: ["apps/web/.env", "apps/api/.env", "config/.env.production"],
});
```

## Generate command flags

CLI flags override config file values when both are provided.

| Flag          | Alias | Description                                    |
| ------------- | ----- | ---------------------------------------------- |
| `--input`     | `-i`  | Path to `.env.example` file(s). Repeatable     |
| `--output`    | `-o`  | Output file path                               |
| `--format`    | `-f`  | Generator(s): `ts`, `zod`, `t3`, `declaration` |
| `--generator` | `-g`  | Alias for `--format`                           |
| `--stdout`    |       | Write output to stdout instead of a file       |
| `--dry-run`   |       | Preview output without writing files           |
| `--no-format` |       | Skip Prettier formatting of output             |
| `--silent`    | `-s`  | Suppress success logs                          |
| `--watch`     | `-w`  | Re-generate on file changes                    |
| `--config`    | `-c`  | Path to config file                            |
| `--version`   | `-v`  | Print version                                  |
| `--help`      | `-h`  | Show help                                      |

## Validation command flags (`check`, `diff`, `doctor`, `verify`)

| Flag                      | Commands                    | Description                                                          |
| ------------------------- | --------------------------- | -------------------------------------------------------------------- |
| `--env <path>`            | `check`, `doctor`, `verify` | Environment source path to validate                                  |
| `--targets <list>`        | `diff`, `doctor`, `verify`  | Comma-separated env targets for drift analysis                       |
| `--contract <path>`       | all                         | Contract file path (`env.contract.mjs` / `.js` / `.ts`)              |
| `--example <path>`        | all                         | Fallback `.env.example` used when no contract file is available      |
| `--strict`                | all                         | Validate extras as errors (default behavior)                         |
| `--no-strict`             | all                         | Downgrade undeclared variables to warnings                           |
| `--json`                  | all                         | Emit compact machine-readable JSON report                            |
| `--json=compact`          | all                         | Explicit alias of `--json` (compact machine-readable JSON)           |
| `--json=pretty`           | all                         | Emit pretty JSON report                                              |
| `--output-file <path>`    | all                         | Persist JSON report to file                                          |
| `--debug-values`          | all                         | Include raw values in issues (unsafe for CI logs)                    |
| `--cloud-provider <name>` | all                         | Add cloud snapshot source: `vercel`, `cloudflare`, `aws`             |
| `--cloud-file <path>`     | all                         | Cloud snapshot JSON file path                                        |
| `--plugin <path>`         | all                         | Plugin module path (repeatable). Module must export a `name` string. |
| `--config <path>`         | all                         | Config file path                                                     |
| `--help`                  | all                         | Show command help                                                    |

## Plugin contract

Plugins are plain ESM modules with a default export matching the `EnvTypegenPlugin` shape:

```typescript
export default {
  name: "my-plugin", // required

  transformSource({ environment, values }) {
    // optional — mutate or replace per-environment values before validation
    return values;
  },

  transformContract(contract) {
    // optional — mutate or replace the loaded contract before validation
    return contract;
  },

  transformReport(report) {
    // optional — post-process the validation report
    return report;
  },
};
```

> The `name` field is **required**. Plugins without it are rejected at startup with a descriptive error.

Register plugins via CLI or config file:

```bash
env-typegen check --env .env --contract env.contract.mjs --plugin ./plugins/my-plugin.mjs
```

```typescript
export default defineConfig({
  plugins: ["./plugins/my-plugin.mjs"],
});
```

## Precedence model

1. CLI flags (highest priority)
2. Config file values
3. Built-in defaults

## Defaults and path resolution

- Strict mode is enabled by default for validation commands.
- `diff` defaults to `.env,.env.example,.env.production` when targets are not provided.
- When `--contract` is omitted, env-typegen auto-discovers `env.contract.mjs`, `env.contract.js`, or `env.contract.ts`.
- All paths are resolved from your current working directory.

## FAQ

### Should I always commit `env-typegen.config.ts`?

Yes for team consistency. It centralizes defaults and reduces CI/local mismatch.

### Can I use different configs per command?

Yes. Use `--config <path>` to load command-specific configuration files.
