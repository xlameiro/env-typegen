## Installation

```bash
pnpm add -D @xlameiro/env-typegen
```

## When to use the API instead of the CLI

- You need custom orchestration in build scripts.
- You want to merge env validation into an existing pipeline tool.
- You want plugin hooks controlled in code.
- You need to process dynamic env content streams before generation.

## Governance command usage from code

Use `runValidationCommand` for all validation commands, including `verify`:

```typescript
const exitCode = await runValidationCommand({
  command: "verify",
  argv: ["--env", ".env", "--contract", "env.contract.mjs", "--json"],
});

if (exitCode !== 0) {
  process.exit(exitCode);
}
```

## Trust, SLO, and forensics outputs

`sync-apply --json` emits governance metadata that can be consumed programmatically:

- `governanceSummary.operationalReadiness` for SLO/incident-derived readiness.
- `governanceSummary.evidence` for signed evidence correlation fields.
- `evidenceBundle.signature` and `evidenceBundle.forensicsIndex` for traceability assertions.

Use these outputs to promote only when trust validation, resiliency checks, and forensics
correlation are all satisfied in CI workflows.

## Core functions

### `parseEnvFile(path)`

Reads a `.env.example` file from disk and returns a `ParsedEnvFile`.

```typescript
const parsed = parseEnvFile(".env.example");
```

### `parseEnvFileContent(content, filePath, options?)`

Parses raw `.env` string content — useful in tests or streaming pipelines.

```typescript
const parsed = parseEnvFileContent("DATABASE_URL=\nPORT=3000", "inline.env");
```

### `inferType(key, value, options?)`

Infers an `EnvVarType` from a key+value pair using heuristics (key name patterns, value shape).

```typescript
const type = inferType("PORT", "3000"); // → "number"
const type2 = inferType("DEBUG", "true"); // → "boolean"
```

### `inferTypes(parsed)`

Runs inference for all variables in a `ParsedEnvFile`.

```typescript
const typed = inferTypes(parsed);
```

### `generateTypeScriptTypes(parsed)`

Produces an exported `EnvVars` type plus `NodeJS.ProcessEnv` namespace augmentation.

```typescript
const output = generateTypeScriptTypes(typed);
```

### `generateZodSchema(parsed)`

Produces a Zod v4 `z.object({ ... })` schema string split into server/client.

```typescript
const output = generateZodSchema(typed);
```

### `generateT3Env(parsed)`

Produces a `@t3-oss/env-nextjs` `createEnv(...)` call string.

```typescript
const output = generateT3Env(typed);
```

### `generateDeclaration(parsed)`

Produces a `.d.ts` file augmenting `NodeJS.ProcessEnv`.

```typescript
const output = generateDeclaration(typed);
```

### `runGenerate(options)`

Orchestrates the full pipeline: parse → infer → generate → write to disk.

```typescript
await runGenerate({
  input: ".env.example",
  output: "env.generated.ts",
  generators: ["typescript", "zod"],
  format: true,
  dryRun: false,
  silent: false,
});
```

### `runValidationCommand({ command, argv })`

Runs validation subcommands programmatically.

```typescript
const exitCode = await runValidationCommand({
  command: "verify",
  argv: ["--env", ".env", "--contract", "env.contract.mjs", "--json"],
});
```

`verify` is intended for CI gates and fails on warnings and errors.

### `loadCloudSource({ provider, filePath })`

Loads environment values from provider snapshot files.

```typescript
const values = await loadCloudSource({
  provider: "aws",
  filePath: "aws-env.json",
});
```

### `loadPlugins({ pluginPaths, configPlugins })`

Loads plugin references and lets you apply hook chains.

```typescript
const plugins = await loadPlugins({
  pluginPaths: ["./plugins/report-plugin.mjs"],
});

const report = applyReportPlugins(baseReport, plugins);
```

## Integration examples

### Build step wrapper

```typescript
await runGenerate({
  input: ".env.example",
  output: "src/env.generated.ts",
  generators: ["typescript", "zod", "declaration"],
  format: true,
  dryRun: false,
  silent: false,

```

### CI validation gate

```typescript
const exitCode = await runValidationCommand({
  command: "verify",
  argv: [
    "--env",
    ".env",
    "--targets",
    ".env,.env.staging,.env.production",
    "--contract",
    "env.contract.mjs",
    "--json",
  ],
});

if (exitCode !== 0) {
  process.exit(exitCode);
}
```

### Pull command integration

Call the CLI subcommand from scripts when provider configuration is already present:

```bash
env-typegen pull vercel --env preview
```

This keeps provider synchronization read-only and separate from validation gate logic.

### Plan and sync-preview command integration

Use CLI invocation from scripts when you need preflight governance decisions:

```bash
env-typegen plan --env .env --contract env.contract.mjs --json
env-typegen sync-preview vercel --env-file .env --config env-typegen.config.mjs --json
```

These commands are read-only and designed for deterministic CI preflight stages.

### Sync-apply command integration

Use `sync-apply` in two stages:

```bash
# dry-run (default)
env-typegen sync-apply vercel --env-file .env --config env-typegen.config.mjs --json

# guarded apply
env-typegen sync-apply vercel \
  --env production \
  --env-file .env \
  --config env-typegen.config.mjs \
  --apply \
  --preflight-file reports/preflight.json \
  --protected-branch \
  --json
```

Apply mode exits non-zero when any write guard blocks execution.

### Policy pack resolution integration

Policy pack precedence is deterministic:

1. Inline policy rules from local config.
2. Overlay packs.
3. Base packs.

Use packs for shared governance defaults and keep project-specific exceptions inline.

### Governance smoke automation

Repository smoke check:

```bash
node qa-test/env-typegen-governance-smoke.mjs
```

The script validates pass/fail verify behavior plus pull/plan/sync-preview execution and writes a machine-readable report.

Apply smoke automation:

```bash
node qa-test/env-typegen-apply-smoke.mjs --mode=all
```

This smoke validates dry-run and apply guardrail outcomes and writes `qa-test/reports/env-governance-apply-smoke.json`.

Promotion smoke command:


- `.github/workflows/env-governance-promotion.yml`

4. Run promotion smoke to validate advisory-enforce -> enforce -> apply progression.

## FAQ

### Is the API stable for automation use?

Yes. The exported functions are intended for programmatic integration, not only CLI internals.

### Should I call generators directly or use `runGenerate`?

Use `runGenerate` for standard pipelines, and direct generator calls when you need custom composition.
