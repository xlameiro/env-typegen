## Why validation matters

Generation alone keeps types in sync, but production issues often come from environment drift.
Validation commands let you enforce contracts before deployment.

## Command decision guide

| Command  | Best when                                          | Output style                        |
| -------- | -------------------------------------------------- | ----------------------------------- |
| `check`  | You want to validate one source against a contract | Pass/fail findings for one source   |
| `diff`   | You need parity across multiple sources            | Drift report across targets         |
| `doctor` | You want one consolidated diagnostic report        | Combined findings + recommendations |
| `verify` | You need a deterministic CI gate                   | Fails on warnings or errors         |

## Contract file

By default, env-typegen auto-discovers `env.contract.mjs` or `env.contract.js`.

```typescript
export default {
  schemaVersion: 1,
  variables: {
    PORT: { expected: { type: "number" }, required: true, clientSide: false },
    NEXT_PUBLIC_APP_URL: {
      expected: { type: "url" },
      required: true,
      clientSide: true,
    },
    APP_VERSION: {
      expected: { type: "semver" },
      required: true,
      clientSide: false,
    },
  },
};
```

If no contract file is found, env-typegen can bootstrap from `.env.example`.

## Usage examples

```bash
# Validate one source
env-typegen check --env .env --contract env.contract.mjs

# Compare drift across standard targets
env-typegen diff --targets .env,.env.example,.env.production --contract env.contract.mjs

# Aggregate diagnostics
env-typegen doctor --env .env --targets .env,.env.example,.env.production --contract env.contract.mjs

# Governance gate for CI
env-typegen verify --env .env --targets .env,.env.production --contract env.contract.mjs
```

## Recommended command lifecycle

1. `check` for local contract correctness.
2. `diff` to compare parity across targets.
3. `doctor` when you want consolidated recommendations.
4. `plan` for policy-aware preflight in read-only mode.
5. `sync-preview` to simulate sync impact in read-only mode.
6. `verify` in CI as the final merge/deploy gate and precondition for `sync-apply`.
7. `conformance` smoke before promotion stage transitions.

## Preflight commands

```bash
env-typegen plan --env .env --contract env.contract.mjs --json=pretty
env-typegen sync-preview vercel --env-file .env --config env-typegen.config.mjs --json
```

Both commands are read-only and intended for governance planning before merge/deploy gates.

## Verify as apply precondition

Before any `sync-apply --apply` execution:

1. Run `verify` and confirm policy decision is acceptable.
2. Run `sync-apply` in dry-run mode first.
3. Execute apply mode only with required preflight and branch protections.

```bash
env-typegen verify --env .env --contract env.contract.mjs --json=pretty
env-typegen sync-apply vercel --env-file .env --config env-typegen.config.mjs --json
```

## Verify in promotion stages

Use `verify` as the stable gate across promotion stages:

1. Advisory-enforce stage: verify policy and artifact consistency without enabling apply writes.
2. Enforce stage: require deterministic verify and dry-run results.
3. Apply stage: allow write path only when verify + guardrails are satisfied.

Promotion smoke command:

```bash
node qa-test/env-typegen-governance-promotion-smoke.mjs
```

Conformance smoke command:

## CI pipeline pattern

```bash
# Pull request and merge gate
env-typegen verify --env .env --contract env.contract.mjs --json=pretty --output-file reports/env-verify.json

# Release drift scan (optional second stage)
env-typegen diff --targets .env,.env.staging,.env.production --contract env.contract.mjs --json=pretty --output-file reports/env-drift.json
```

By design, `verify` exits with code `1` when any warning or error exists.

Recommended governance policy:

- Run `verify` as a blocking gate for PRs to `main` and protected branches.
- For feature branches, teams may keep the same check as non-blocking while onboarding.

## Strict mode

- Strict mode is enabled by default.
- Use `--no-strict` to downgrade undeclared variables to warnings.

## Cloud snapshot sources

Use provider snapshots as validation inputs:

```bash
env-typegen check --cloud-provider vercel --cloud-file vercel-env.json --contract env.contract.mjs
env-typegen diff --cloud-provider cloudflare --cloud-file cloudflare-env.json --contract env.contract.mjs
env-typegen doctor --cloud-provider aws --cloud-file aws-env.json --contract env.contract.mjs
```

Supported providers: `vercel`, `cloudflare`, `aws`.

Provider sync remains read-only in v1. Use `pull` to inspect and normalize provider state without remote writes.

## Plugins

Plugins can transform contract, source values, and final report.

```typescript
export default {
  name: "report-plugin",
  transformReport: (report) => ({
    ...report,
    recommendations: [...(report.recommendations ?? []), "plugin-applied"],
  }),
};
```

Load them via `--plugin` flags or `plugins` in `env-typegen.config.mjs`.

All validation paths (`--env`, `--contract`, `--targets`, `--cloud-file`, `--output-file`) are resolved from your current working directory.
For direct Node.js runtime execution, prefer `.mjs` or `.js` contract files.
TypeScript contract files (`.ts`) require a runtime loader such as `tsx` or `ts-node`.

## FAQ

### Should I run `check` and `diff`, or only `doctor`?

Use `doctor` when you want one command for diagnostics. Use `check` and `diff` separately when your CI has stage-specific gates.

### Which command should block merge in CI?

Use `verify`. Keep `check`, `diff`, and `doctor` for diagnostics and staged pipelines.

### Can I include sensitive values in JSON reports?

Only when intentionally debugging. Use `--debug-values` with care because values may leak into logs.

### Do cloud snapshots replace local `.env` checks?

No. They complement local checks and help detect cross-environment drift before rollout.

### Is there an operational smoke check for governance flows?

Yes. Run `node qa-test/env-typegen-governance-smoke.mjs` to validate verify/pull/plan/sync-preview behavior and generate `qa-test/reports/env-governance-smoke.json`.

### Is there a smoke check for apply guardrails?

Yes. Run `node qa-test/env-typegen-apply-smoke.mjs --mode=all` to validate dry-run/apply pass/fail behavior and generate `qa-test/reports/env-governance-apply-smoke.json`.

### Is there a conformance smoke check for adapter/orchestration behavior?

Yes. Run `node qa-test/env-typegen-conformance-smoke.mjs` to validate adapter contract v3 and bounded orchestration semantics.

### Is there a forensics smoke check for signed evidence integrity?

Yes. Run `node qa-test/env-typegen-forensics-smoke.mjs` to validate signed evidence,
forensics chain integrity, and governance evidence correlation.
