## Installation

```bash
pnpm add -D @xlameiro/env-typegen
```

## 1) Generate your first typed output

```bash
env-typegen --input .env.example --output env.generated.ts --format ts
```

This command gives you a typed environment contract directly from `.env.example`.

## 2) Generate multiple outputs from one source

```bash
env-typegen -i .env.example -o env.generated.ts -f ts -f zod -f t3 -f declaration
```

When multiple formats are specified, each generator writes a separate file:

- `env.generated.typescript.ts`
- `env.generated.zod.ts`
- `env.generated.t3.ts`
- `env.generated.declaration.d.ts`

## 3) Enable watch mode during local development

Automatically re-generates on every change to your `.env.example`:

```bash
env-typegen -i .env.example -o env.generated.ts -f ts --watch
```

## 4) Create a validation contract

Create `env.contract.mjs` in your project root:

```typescript
export default {
  schemaVersion: 1,
  variables: {
    PORT: { expected: { type: "number" }, required: true, clientSide: false },
    NEXT_PUBLIC_APP_URL: {
      expected: { type: "url" },
      required: true,
      clientSide: true,
    },
  },
};
```

## 5) Validate environment files against that contract

```bash
# Validate one environment source
env-typegen check --env .env --contract env.contract.mjs

# Compare drift across multiple sources
env-typegen diff --targets .env,.env.example,.env.production --contract env.contract.mjs

# Aggregate and prioritize diagnostics
env-typegen doctor --env .env --targets .env,.env.example,.env.production --contract env.contract.mjs

# Deterministic CI gate
env-typegen verify --env .env --targets .env,.env.production --contract env.contract.mjs
```

`verify` is the governance gate command. It returns exit code `1` when warnings or errors exist.

## 6) Pull provider state (read-only)

Add providers in your config and pull by logical environment:

```bash
env-typegen pull vercel --env preview
```

Use `pull` to inspect and normalize remote state before drift checks. It does not write back to the provider.

## 7) Export JSON output for CI pipelines

```bash
env-typegen verify --env .env --json=pretty --output-file reports/env-verify.json
```

For direct Node.js runtime execution, prefer `.mjs` or `.js` for contract files.
TypeScript contract files (`.ts`) require a runtime loader such as `tsx` or `ts-node`.

Strict mode is enabled by default for validation commands. Use `--no-strict` to downgrade undeclared variables to warnings.

All command paths (`--env`, `--contract`, `--targets`, and `--output-file`) are resolved from your current working directory.

## Recommended rollout path by maturity

1. Start with generation-only (`ts` + `zod`) to remove manual typing.
2. Add `check` in CI to enforce required variables and types.
3. Add `pull` and provider config to inspect remote state safely.
4. Add `verify` as the merge-blocking governance gate.
5. Add `diff`, `doctor`, cloud snapshots, and plugins when your team needs deeper diagnostics.

## FAQ

### Do I need all generators from day one?

No. You can start with one format and add more over time with repeated `--format` flags.

### What if I do not have `env.contract.mjs` yet?

env-typegen can bootstrap from `.env.example`, but explicit contracts are recommended for governance.
